Run `pod outdated` to check which CocoaPods dependencies in your project have newer versions available. This compares the currently installed versions (from `Podfile.lock`) against the latest versions in the CocoaPods spec repository, showing you what updates are available without making any changes. Use this to plan dependency updates and stay on top of security patches.

## Parameters

| Parameter | Type | Required | Default | Description |
|-----------|------|----------|---------|-------------|
| `projectPath` | string | No | — | Absolute path to the directory containing the `Podfile`. If omitted, uses the current working directory. |

## Usage Examples

### Check for outdated pods

> "Which of my CocoaPods have newer versions?"

```json
{
  "projectPath": "/Users/johndoe/Projects/MyApp"
}
```

### Audit dependencies before a release

> "Check if any pods are outdated before we cut the release"

```json
{
  "projectPath": "/Users/johndoe/Projects/MyApp"
}
```

## Tips

<Aside type="tip">
  Run `pod_outdated` regularly (e.g., weekly or before each release) to stay on top of dependency updates, especially security patches.
</Aside>

<Aside type="tip">
  After identifying outdated pods, use `pod_update` with the `podName` parameter to update them one at a time, testing your app between each update to isolate any issues.
</Aside>

<Aside type="note">
  This tool only reports what is available. It does not modify `Podfile.lock` or install anything. Use `pod_update` to actually perform the updates.
</Aside>

<Aside type="note">
  The "latest" version shown may exceed your `Podfile` version constraints. If so, you would need to update the version constraint in your `Podfile` first before running `pod_update`.
</Aside>
